== Current Status: Prototype (v0.1.0)

V core application and ReScript browser extension compile successfully.
No real audio I/O, no tests, no functional end-to-end pipeline.
See `TESTING-REPORT.adoc` for build verification details.

== Architecture Decisions

All architecture decisions are documented in `docs/DESIGN-DECISIONS.adoc` and codified in `.machine_readable/META.scm`.

Key decisions:

* **Hybrid AGC + compressed range limiter** (not statistical/LLM)
* **Lookahead processing** (50-100ms pre-emptive adjustment)
* **RMS as proxy for LUFS** (psychophysically informed, computationally cheap)
* **Loopback audio capture** (outgoing signal stage via PipeWire monitor sink)
* **Linux-first** (PipeWire on Fedora Kionite, then Windows/macOS)
* **On-device only** (no cloud, no telemetry)

=== v0.2.0 - Audio Pipeline

_Target: 2026 Q2_

Priority: Get real audio flowing through the processing chain.

* [ ] **PipeWire/PulseAudio integration** - Loopback audio capture on Linux
** Evaluate V bindings for PipeWire/PulseAudio
** If V bindings insufficient, implement Rust shim using `cpal` + `pulsectl` crates
** Wire loopback capture to processor chain
* [ ] **Lookahead buffer** - Circular buffer with 50-100ms lookahead window
** Dedicated audio capture thread
** Asynchronous stream handler
** Lookahead index ahead of playback position
* [ ] **RMS calculation** - Real-time loudness measurement
** Per-buffer RMS computation
** Running average with configurable window
* [ ] **Volume control** - System volume adjustment via PipeWire/PulseAudio API
* [ ] **Unit tests** - Core module test coverage
** `audio_buffer_test.v`
** `dsp_utils_test.v`

== Critical Path

The critical path to MVP (v0.3.0) requires:

. Resolve language decision for audio capture backend (V vs. Rust shim)
. Implement PipeWire loopback capture
. Implement lookahead buffer
. Wire audio pipeline end-to-end
. System tray UI operational

Everything else can proceed in parallel.
